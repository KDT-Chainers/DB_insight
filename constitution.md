# constitution.md

## 1. 역할 정의

당신은 이 프로젝트에서 작업을 수행하는 코딩 에이전트이다.  
이 문서는 모든 작업에서 반드시 따라야 하는 최상위 규칙이다.  
모든 판단과 구현은 이 문서를 기준으로 이루어져야 한다.

이 문서는 항상 로드되는 1계층이며, 어떤 Knowledge 문서보다 우선한다.

---

## 2. 시스템 구조 이해

이 프로젝트는 다음 3계층 구조를 따른다.

- Constitution (1계층): 모든 작업의 기준 규칙
- Agents (2계층): 역할별 전문 작업 수행 주체
- Knowledge (3계층): 최소한의 공통 스펙

당신은 항상 이 구조를 전제로 작업해야 한다.

---

## 3. 작업 원칙

### 3.1 역할 기반 수행

당신은 자신의 역할(agent.md)에 정의된 범위 내에서만 작업해야 한다.

- frontend-agent → UI/UX
- backend-agent → 서버 및 로직
- database-agent → 데이터 구조
- embedding-agent → 파일 처리 및 임베딩

다른 영역의 작업이 필요할 경우, 해당 영역의 규칙을 따른다.

---

### 3.2 지식 사용 우선순위

작업 시 다음 순서를 반드시 따른다.

1. constitution.md (항상 우선)
2. 자신의 agent.md (전문 지식)
3. Knowledge 문서 (필요 시만 참조)

Knowledge 문서는 부족할 때만 참고하며, 기본 로직은 agent에 포함되어 있어야 한다.

---

### 3.3 중복 구현 금지

- 동일한 로직을 여러 위치에 중복 구현하지 않는다
- 데이터 구조, API, 검색 방식은 항상 하나의 기준만 따른다
- 기준이 필요할 경우 Knowledge 문서를 참고한다

---

## 4. 데이터 흐름 규칙

모든 데이터는 다음 순서를 반드시 따른다.

raw_DB → extracted_DB → embedded_DB

이 순서를 위반하는 구현은 금지된다.

### 규칙

- raw_DB 없이 extracted_DB 생성 금지
- extracted_DB 없이 embedded_DB 생성 금지
- 각 단계는 이전 단계의 결과를 기반으로 생성되어야 한다

---

## 5. 식별자 규칙

모든 데이터는 일관된 식별자를 사용해야 한다.

### file_id

- 모든 파일의 고유 식별자
- raw_DB에서 생성
- 모든 계층에서 동일하게 사용

### chunk_id

- extracted_DB와 embedded_DB에서 사용
- 반드시 file_id를 포함해야 한다

예시 형태:

file_id: file_001  
chunk_id: file_001_chunk_001

---

## 6. 데이터 일관성 규칙

다음 규칙을 반드시 지켜야 한다.

- file_id는 모든 DB에서 동일해야 한다
- chunk 기준은 extracted와 embedded에서 동일해야 한다
- metadata 구조는 일관되어야 한다

---

## 7. 인덱싱 및 검색 규칙

### 인덱싱

- 모든 파일은 동일한 기준으로 chunking되어야 한다
- 임베딩 방식은 IndexingSpec 기준을 따른다

### 검색

- 검색은 반드시 embedded_DB 기준으로 수행한다
- raw_DB나 extracted_DB를 직접 검색하지 않는다
- 검색 결과는 metadata를 통해 원본 파일과 연결한다

---

## 8. API 및 데이터 계약

- 프론트와 백엔드는 API.md를 기준으로 통신한다
- 데이터 구조는 DataContract.md를 기준으로 통일한다
- 임의의 필드 추가/변경은 금지된다

---

## 9. 오류 처리 원칙

- 모든 오류는 무시하지 않고 처리한다
- 실패 시 원인을 추적할 수 있어야 한다
- 데이터 손실이 발생하는 처리는 금지된다

---

## 10. 금지 사항

다음 행동은 절대 금지된다.

- 데이터 흐름 단계 생략
- file_id / chunk_id 규칙 위반
- Knowledge 없이 임의 구조 생성
- 역할 범위를 벗어난 구현
- 동일 기능 중복 구현

---

## 11. 최종 목표

당신의 목표는 다음과 같다.

- 일관된 구조 유지
- 재현 가능한 데이터 처리
- 안정적인 검색 결과 제공

모든 구현은 “작동”이 아니라  
“일관되고 유지 가능한 구조”를 목표로 해야 한다.
