1router_node — 질문 유형 분류

🧭
Step 1 · Router
질문 유형 판별
LLM에게 질문을 보여주고 4가지 중 하나로 분류합니다.
분류 결과에 따라 이후 파이프라인이 완전히 달라집니다.
📡 SSE: route
🔍 rag
파일에서 정보를 찾아야 할 때
예) "삼성 재생에너지 전환율이 몇 %야?"
💬 chat
파일 없이 일반 대화할 때
예) "파이썬 리스트 정렬 방법 알려줘"
↩️ followup
이전 턴 파일을 그대로 쓸 때
예) "그러면 그 파일에서 3장도 알려줘"
📝 qa_gen
퀴즈·문제 생성 요청할 때
예) "이 문서로 퀴즈 만들어줘"
⚠️ 주의: "FAO 가격지수가 얼마야?"처럼 질문 형태인데도 qa_gen으로 오분류되는 버그가 있었음.
→ 프롬프트에 "명시적 생성 요청이 없으면 무조건 rag" 규칙 추가 + 테스트마다 고유한 thread_id 사용으로 해결.

2intent_node — 의도 파악 + 키워드 추출

🎯
Step 2 · Intent
의도 요약 + 검색 키워드 생성
LLM이 질문을 분석해 3가지 정보를 추출합니다.
📡 SSE: intent

출력 3가지

항목예시용도
의도메시지 "삼성전자 탄소중립 목표와 재생에너지 전환율을 문서에서 찾아드릴게요." UI에 표시
파일검색 키워드 ["삼성전자", "탄소중립", "재생에너지"] 벡터 DB 검색
내용검색 키워드 ["재생에너지", "탄소중립"] 파일 내 본문 스캔

LLM 프롬프트 형식

질문: 삼성전자 재생에너지 전환율은?
의도메시지: 삼성전자 재생에너지 전환율을 찾아드릴게요.
파일검색: 삼성전자 재생에너지
내용검색: 재생에너지 전환율
※ 형식을 벗어난 출력이 오면 질문에서 직접 명사 추출 (Fallback)

3search_node — 벡터 DB 후보 검색

🔍
Step 3 · Search
벡터 유사도 검색
파일검색 키워드를 임베딩 벡터로 변환 후, DB에 저장된 파일 벡터들과 코사인 유사도를 계산해
상위 topK개 파일을 후보로 선정합니다. UI에는 파일 카드로 즉시 표시됩니다.
📡 SSE: candidates
왜 원본 파일 전체를 검색하지 않고 벡터를 쓰나?
179,328자짜리 PDF를 통째로 하나의 벡터로 만들면 세부 수치(93.4%, 3,035.5만 톤)가 평균화되어 희석됩니다. 청크로 나눠 각각 임베딩하면 관련 단락의 벡터가 질문 벡터에 더 가까워져 정확한 파일을 찾을 수 있습니다.

4scan_node — 파일 본문 정밀 스캔

📡
Step 4 · Scan
키워드 슬라이딩 윈도우 스캔
후보 파일 각각을 fitz(PyMuPDF)로 직접 읽어 전문 텍스트를 추출한 뒤,
내용검색 키워드가 나타나는 위치를 찾고 앞뒤 ±400자를 window로 추출합니다.
키워드가 하나라도 발견되면 found, 없으면 not_found.
📡 SSE: scanning (파일마다) 📡 SSE: scan_result (found/not_found)

왜 청크에서 검색하나? (원본 전체 vs 청크)

📏
컨텍스트 한계
LLM에 179K자 전체를 넣으면 중요한 수치를 긴 문서 속에서 놓치는 "바늘 찾기(Needle in Haystack)" 문제 발생. 핵심 단락만 추려야 정확도가 올라간다.
🎯
노이즈 제거
scan_node의 ±400자 윈도우는 이미 관련 단락으로 정제된 덩어리. 전체를 통째로 넘기는 것보다 정확한 수치 포함 확률이 높다.
🔤
fitz 줄바꿈 문제
PDF를 읽으면 단어 중간에 \n이 들어감. 예: "재생에너지\n전환율은 93.4%". _join_pdf_lines()로 소프트 줄바꿈을 제거해야 키워드 매칭이 성공한다.
🌏
중국어 필터
삼성 ESG 보고서에 중국어 섹션 혼입 → CJK 문자 비율 25% 이상인 줄은 자동 제거. 답변에 중국어가 섞이는 문제 해결.

5select_node — 최종 소스 확정

✅
Step 5 · Select
발견된 파일만 추려냄
scan_node에서 found 판정을 받은 파일만 matched_sources로 확정합니다.
이 파일들이 generate_node의 컨텍스트 소스가 됩니다.
📡 SSE: selected

6generate_node — 답변 생성 (핵심)

✨
Step 6 · Generate
컨텍스트 조립 → key_facts 추출 → LLM 답변
가장 복잡한 단계입니다. 크게 3단계로 나뉩니다.
📡 SSE: key_facts 📡 SSE: generating 📡 SSE: token (스트리밍) 📡 SSE: done

6-1. 컨텍스트 조립

matched_sources 각각에 대해:

① scan_chunks        ← scan_node가 찾아준 ±400자 윈도우 (이미 정제됨)
② fitz head[:5000]  ← 문서 앞 5000자 (메타, 총괄 수치 포함)
③ keyword_target     ← 나머지 구간에서 키워드 슬라이딩 검색

combined = ①  +  ②  +  ③  (최대 15,000자)

6-2. key_facts 추출 (핵심 수치 강제 고정)

LLM이 학습 데이터 기억으로 환각하는 것을 막기 위해, Python regex로 문서에서 직접 수치 포함 문장을 추출합니다.

scan_chunks
min_score=1 (관대)
+
fitz_head[:3000]
min_score=4 (엄격)
→
중복 제거
최대 6개
→
key_facts 확정
UI + 프롬프트 이중 노출 
def _python_extract_key_facts(full_text, question, max_facts=6, min_score=1):
    # 1. 문장 분리 (줄바꿈 + 문장 종결 어미 기준)
    # 2. score = 숫자 패턴 수 × 3 + 질문 키워드 매칭 수
    # 3. score ≥ min_score AND 키워드 1개 이상 포함한 문장만
    # 4. 점수 내림차순 정렬 → 상위 max_facts개 반환
왜 fitz_head에 min_score=4를 쓰나?
FAO PDF 앞부분(식량가격지수 섹션)에는 107.3 같은 숫자가 많은데, Q2(곡물 생산량) 질문과 키워드가 겹쳐 잘못된 수치가 key_facts에 들어가는 오염이 발생했었음. min_score=4로 엄격히 필터하면 관련도 낮은 수치는 걸러진다.

6-3. 이중 노출(Forced-Quote) 프롬프트

key_facts를 시스템 프롬프트 + 유저 메시지 양쪽에 모두 삽입합니다.
Gemma3:12b처럼 IFEval 점수가 높은 모델은 이 규칙을 따라 학습 기억을 무시하고 문서 수치를 그대로 씁니다.

2. 학습 데이터에서 알고 있는 수치를 쓰면 안 됩니다.
...

--- 유저 메시지 ---
[문서 핵심 인용]

위 인용문의 수치를 그대로 사용해서 다음 질문에 답하세요:
삼성전자의 2030년 탄소중립 목표와 2024년 재생에너지 전환율은?

7모델 선택 — 왜 Gemma3:12b인가?

RAG에서 환각이 발생하는 이유

RAG는 "이 문서만 봐"라고 지시하는 방식인데, 작은 모델은 유명한 정보에 대한 학습 prior가 너무 강해서 문서를 눈앞에 줘도 학습 기억을 꺼냅니다.

삼성 ESG는 공개된 유명 보고서 → Qwen7b 학습 데이터에 이미 포함 → "내가 아는 게 50%/100%인데 문서가 93.4%라고?" → 문서 무시 → 환각
모델 디스크 VRAM IFEval↑ Q4 (93.4%) RTX 4060
Qwen2.5:7b 4.7GB 5.5GB 71.2 ❌ 100% 환각 ✅
Qwen2.5:14b 9GB 10GB 81.0 — ❌ VRAM 초과
Gemma3:12b ✓ 8.1GB ~8.5GB 88.9 ✅ 93.4% 정답 ⚠️ CPU offload
Gemma3:27b 17GB 17GB 90.4 — ❌ 불가

IFEval = "지시 준수 능력" 벤치마크. 높을수록 시스템 프롬프트 규칙을 잘 따름 → RAG 정확도의 핵심 지표.

8UI 실시간 업데이트 흐름

백엔드 각 노드가 완료될 때마다 SSE 이벤트를 보내고, 프론트엔드가 즉시 UI를 업데이트합니다.

📡 route
라우팅 결과 수신 → RAG · 파일 검색 뱃지 표시
📡 intent
의도메시지 + 키워드 수신 → "삼성전자 탄소중립 목표와 재생에너지 전환율을 문서에서 찾아드릴게요." + 키워드 태그 표시
📡 candidates
후보 파일 목록 수신 → 오른쪽 패널에 파일 카드 즉시 표시
📡 scanning / scan_result
파일별로 스캔 진행 → 왼쪽 스캔 로그에 스캔 중 / 발견됨 / 없음 애니메이션. 파일 카드에도 상태 오버레이 표시
📡 selected
최종 소스 확정 → 출처 목록 업데이트
📡 key_facts (신규)
Python이 추출한 핵심 인용문 → 📌 핵심 인용 섹션으로 초록 카드에 표시
📡 generating (신규)
LLM 생성 시작 → "답변 생성 중…" 스피너 표시
📡 token × N
답변 토큰 스트리밍 → 글자가 실시간으로 하나씩 추가됨. 생성 중 스피너 사라짐
📡 done
생성 완료 → 커서 블링크 사라짐, 출처 링크 클릭 가능

9알려진 한계 & 개선 방향

Q1 원인 파악 미흡

FAO 가격지수 128.5는 맞추지만, 유지류/설탕 상승 원인을 못 찾음.

원인: 질문에 "팜유", "말레이시아", "해바라기유", "에탄올", "브라질" 키워드가 없어 scan_node가 해당 단락을 아예 찾지 못함.
→ 개선: 질문 키워드 → 관련 키워드 자동 확장(Query Expansion) 로직 필요

Q3 이해관계자 채널 환각

"Slack", "월례 보고회" 등 문서에 없는 내용 출력.

원인: Samsung ESG 이해관계자 소통 채널이 표(Table) 형태로 PDF에 있어 fitz가 평탄화하면서 맥락 손실.
→ 개선: PDF 표 구조 파싱(pdfplumber 등) + 표 행·열 재구성
